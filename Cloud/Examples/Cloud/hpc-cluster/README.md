## 1. What This Project Does

This Terraform project provisions a minimal but fully functional HPC cluster on an OpenStack cloud (such as CINECA's ADA Cloud) from scratch. Running a single command (`terraform apply`) creates:

- a **private virtual network** with a dedicated subnet and router
- a **master node** that acts simultaneously as login node, SLURM controller, and NFS server
- two **compute nodes** connected only to the internal network, invisible from the internet
- a **shared filesystem** (`/shared`) exported from the master via NFS and mounted on every compute node
- a complete **SLURM** installation across all nodes, including munge authentication
- an **SSH keypair** registered in OpenStack

The entire cluster is configured automatically at boot time using cloud-init shell scripts generated by Terraform. No manual intervention is required after `terraform apply`.

This project is intentionally minimal. It does not use Ansible, Chef, or Puppet. Everything is done with standard shell scripts inside cloud-init `user_data`, which makes the logic completely transparent and easy to study line by line.

## 2. Cluster Architecture

```
Internet
    │
    │  SSH on port 22  (floating IP only on master)
    ▼
┌─────────────────────────────────────────────────────────────────────┐
│  OpenStack Project                                                  │
│                                                                     │
│  External network  (provider network, routed to internet)           │
│         │                                                           │
│      [Router]  ←── NAT gateway to external network                  │
│         │                                                           │
│  Internal network: 192.168.100.0/24                                 │
│                                                                     │
│  ┌──────────────────────────────────────────┐                       │
│  │  master   192.168.100.10                 │                       │
│  │  ├── slurmctld  (SLURM controller)       │ ←── floating IP       │
│  │  ├── slurmd     (can run jobs too)       │     (public IP)       │
│  │  ├── NFS server exports /shared          │                       │
│  │  └── munge daemon (authentication)       │                       │
│  └──────────┬────────────────────┬──────────┘                       │
│             │  NFS + SLURM       │                                  │
│      ┌──────┴──────┐      ┌──────┴──────┐                           │
│      │  node01     │      │  node02     │                           │
│      │  .100.11    │      │  .100.12    │                           │
│      │  slurmd     │      │  slurmd     │   no floating IP          │
│      │  /shared    │      │  /shared    │   (internal only)         │
│      │  (NFS)      │      │  (NFS)      │                           │
│      └─────────────┘      └─────────────┘                           │
└─────────────────────────────────────────────────────────────────────┘
```

**Key networking points**:

- The master is the only node with a **floating IP** (a public, routable IP address). This is your SSH entry point into the cluster.
- The compute nodes have only **private IPs** on the internal subnet. They are not directly reachable from the internet. You access them via SSH through the master (jump host pattern).
- All three nodes can reach the internet via the router's NAT gateway. This is needed during cloud-init to run `apt-get install`.
- SLURM, NFS, and munge traffic all flows over the internal `192.168.100.0/24` network.

## 3. How Terraform Orchestrates the Deployment

Terraform reads all `.tf` files, builds a dependency graph of every resource, and creates them in the correct order. The deployment sequence for this project is:

```
1.  provider "openstack"                     reads OS_* environment variables (openrc.sh)
2.  data sources                             queries OpenStack for existing image + external net
3.  openstack_compute_keypair_v2             uploads your public SSH key
4.  openstack_networking_network_v2          creates the internal virtual network
5.  openstack_networking_subnet_v2           adds subnet 192.168.100.0/24
6.  openstack_networking_router_v2           creates a virtual router
7.  openstack_networking_router_interface_v2 connects subnet to router (enables internet)
8.  openstack_networking_secgroup_v2 (×2)   creates master and compute security groups
9.  openstack_networking_secgroup_rule_v2    adds firewall rules to each security group
10. openstack_networking_port_v2 (master)   Neutron port with fixed IP .10
11. openstack_networking_port_v2 (compute×2)Neutron ports with fixed IPs .11, .12
12. openstack_networking_floatingip_v2      allocates a floating IP from the external pool
13. openstack_compute_instance_v2 (master)  boots master VM — cloud-init starts in background
14. openstack_compute_floatingip_associate_v2 attaches floating IP to master port
15. openstack_compute_instance_v2 (×2)     boots compute VMs — cloud-init starts in background
```

Steps 13–15 involve VM boot and cloud-init execution. Terraform reports success as soon as the Nova API accepts the boot request, but cloud-init runs in the background for several more minutes. The cluster becomes fully operational 3–8 minutes after `terraform apply` completes.

## 4. File-by-File Reference

### 4.1 `main.tf`

This file contains the provider declaration, data sources, SSH keypair, and all **network infrastructure**.

**Provider block**:
```hcl
terraform {
  required_version = ">= 1.5.0"
  required_providers {
    openstack = {
      source  = "terraform-provider-openstack/openstack"
      version = "~> 2.0"
    }
  }
}

provider "openstack" {}
```

The provider block is intentionally empty. Terraform reads all connection details from environment variables populated by `source openrc.sh`:

| Environment variable | Meaning |
|---|---|
| `OS_AUTH_URL` | Keystone authentication endpoint |
| `OS_USERNAME` | Your OpenStack username |
| `OS_PASSWORD` | Your password |
| `OS_PROJECT_NAME` | The project/tenant to deploy into |
| `OS_REGION_NAME` | Region (e.g. `RegionOne`) |

Never put credentials directly in `.tf` files — they would end up in version control.

**Data sources** query OpenStack for existing resources that Terraform does not manage but needs to reference:

```hcl
data "openstack_networking_network_v2" "external" {
  name = var.external_network_name   # e.g. "external"
}

data "openstack_images_image_v2" "ubuntu" {
  name        = var.image_name       # e.g. "Ubuntu 22.04"
  most_recent = true
}
```

The external network is the provider network managed by the cloud operator. The cluster router uses it as its upstream gateway. The image data source finds the VM boot image by name.

**Network resources** build the cluster's private network layer by layer:

```
openstack_networking_network_v2          the L2 network (like a virtual switch)
openstack_networking_subnet_v2           L3 subnet with CIDR and DHCP
openstack_networking_router_v2           virtual router (connected to external network)
openstack_networking_router_interface_v2 connects the subnet to the router
```

This four-resource pattern is standard in OpenStack Neutron. Without the router interface, VMs could not reach the internet (needed for `apt-get`) and floating IPs would not be routable.

**Security groups** are stateful firewall rules applied to network ports:

```hcl
# Master accepts SSH from anywhere, plus all traffic from the internal subnet
resource "openstack_networking_secgroup_rule_v2" "master_ssh" {
  direction        = "ingress"
  protocol         = "tcp"
  port_range_min   = 22
  port_range_max   = 22
  remote_ip_prefix = "0.0.0.0/0"
  ...
}

resource "openstack_networking_secgroup_rule_v2" "master_internal" {
  direction        = "ingress"
  remote_ip_prefix = var.internal_subnet_cidr   # 192.168.100.0/24
  ...
}

# Compute nodes accept only traffic from the internal subnet
resource "openstack_networking_secgroup_rule_v2" "compute_internal" {
  direction        = "ingress"
  remote_ip_prefix = var.internal_subnet_cidr
  ...    
}
```

Egress (outbound) traffic is allowed by default in OpenStack. Security groups are attached to **Neutron ports**, not to instances directly. This is the correct approach in modern OpenStack: attaching security groups to instances via the Nova API has a known bug where the rules are not applied until after the first state refresh.

### 4.2 `variables.tf`

This file declares all configurable parameters. You customise the cluster by editing `terraform.tfvars`, not the `.tf` files themselves. Every variable has a type, description, and default value.

| Variable | Default | What it controls |
|---|---|---|
| `cluster_name` | `"mhpc"` | Prefix for all resource names (VMs, ports, security groups) |
| `external_network_name` | `"external"` | Provider network for floating IPs — check with `openstack network list --external` |
| `internal_subnet_cidr` | `"192.168.100.0/24"` | Private address space for the cluster |
| `dns_nameservers` | `["8.8.8.8","8.8.4.4"]` | DNS given to VMs via DHCP |
| `master_ip` | `"192.168.100.10"` | Fixed private IP of the master node |
| `compute_ip_base` | `11` | Last octet base for compute nodes: node01=.11, node02=.12, ... |
| `image_name` | `"Ubuntu 22.04"` | VM image name — must match exactly what is in your OpenStack project |
| `master_flavor` | `"m1.medium"` | Flavor (VM size) for the master node |
| `compute_flavor` | `"m1.medium"` | Flavor for compute nodes |
| `compute_node_count` | `2` | Number of compute nodes to create |
| `keypair_name` | `"mhpc-key"` | Name under which your public key is registered in OpenStack |
| `public_key_path` | `"~/.ssh/id_rsa.pub"` | Path to your local public key file |
| `shared_fs_path` | `"/shared"` | NFS export and mount point on all nodes |
| `compute_cpus` | `2` | Must match the vCPU count of `compute_flavor` — written into `slurm.conf` |
| `compute_mem_mb` | `3500` | Must be slightly below actual RAM of `compute_flavor` — written into `slurm.conf` |

The last two (`compute_cpus`, `compute_mem_mb`) are critical. If you tell SLURM a node has 8 CPUs but the VM only has 2, SLURM will overcommit resources and jobs will fail in confusing ways.

### 4.3 `master.tf`

This file defines three resources: the Neutron port, the master VM instance, and the floating IP attachment.

**Neutron port with a fixed IP**:
```hcl
resource "openstack_networking_port_v2" "master" {
  name               = "${var.cluster_name}-master-port"
  network_id         = openstack_networking_network_v2.cluster.id
  security_group_ids = [openstack_networking_secgroup_v2.master.id]

  fixed_ip {
    subnet_id  = openstack_networking_subnet_v2.cluster.id
    ip_address = var.master_ip   # 192.168.100.10
  }
}
```

A Neutron port is the virtual NIC. By creating it explicitly with `fixed_ip`, we guarantee the master always gets the same private IP regardless of DHCP ordering. This is essential because `slurm.conf` references the master by IP (`SlurmctldHost=master(192.168.100.10)`), and compute nodes hard-code the master's IP into `/etc/hosts` during cloud-init. If the IP changed after a reboot, the cluster would break.

**The master instance and its cloud-init script**:
```hcl
resource "openstack_compute_instance_v2" "master" {
  name        = "${var.cluster_name}-master"
  image_id    = data.openstack_images_image_v2.ubuntu.id
  flavor_name = var.master_flavor
  key_pair    = openstack_compute_keypair_v2.cluster.name

  network {
    port = openstack_networking_port_v2.master.id
  }

  user_data = templatefile("${path.module}/templates/master_init.sh.tpl", {
    cluster_name   = var.cluster_name
    master_ip      = var.master_ip
    shared_path    = var.shared_fs_path
    node_count     = var.compute_node_count
    compute_cpus   = var.compute_cpus
    compute_mem_mb = var.compute_mem_mb
    node_ips = [
      for i in range(var.compute_node_count) :
      cidrhost(var.internal_subnet_cidr, var.compute_ip_base + i)
    ]
  })
}
```

The `user_data` argument passes a rendered shell script to cloud-init, which executes it as root on first boot. The `cidrhost()` function computes IP addresses: `cidrhost("192.168.100.0/24", 11)` returns `"192.168.100.11"`. The `for` expression generates the full list of compute node IPs automatically.

**Floating IP allocation and association**:
```hcl
resource "openstack_networking_floatingip_v2" "master" {
  pool = var.external_network_name
}

resource "openstack_compute_floatingip_associate_v2" "master" {
  floating_ip = openstack_networking_floatingip_v2.master.address
  instance_id = openstack_compute_instance_v2.master.id
  fixed_ip    = openstack_networking_port_v2.master.fixed_ip[0].ip_address
}
```

A floating IP is a public address from the external pool that OpenStack's network layer translates to the master's private IP (`192.168.100.10`) before delivering packets to the VM. Specifying `fixed_ip` in the association avoids ambiguity when a VM has multiple network interfaces.

### 4.4 `compute.tf`

This file creates all compute nodes using the `count` meta-argument, which tells Terraform to create multiple identical resources:

```hcl
resource "openstack_networking_port_v2" "compute" {
  count = var.compute_node_count

  name       = "${var.cluster_name}-node0${count.index + 1}-port"
  fixed_ip {
    ip_address = cidrhost(var.internal_subnet_cidr, var.compute_ip_base + count.index)
  }
  ...
}
```

`count.index` is 0-based. For `compute_node_count = 2`:
- index 0 → name `mhpc-node01-port`, IP `192.168.100.11`
- index 1 → name `mhpc-node02-port`, IP `192.168.100.12`

The VM instances follow the same pattern:
```hcl
resource "openstack_compute_instance_v2" "compute" {
  count       = var.compute_node_count
  name        = "${var.cluster_name}-node0${count.index + 1}"

  user_data = templatefile("${path.module}/templates/compute_init.sh.tpl", {
    node_name    = "node0${count.index + 1}"
    node_ip      = cidrhost(var.internal_subnet_cidr, var.compute_ip_base + count.index)
    master_ip    = var.master_ip
    shared_path  = var.shared_fs_path
    all_node_ips = [
      for i in range(var.compute_node_count) :
      cidrhost(var.internal_subnet_cidr, var.compute_ip_base + i)
    ]
  })

  depends_on = [
    openstack_compute_instance_v2.master,
    openstack_compute_floatingip_associate_v2.master,
  ]
}
```

The `depends_on` is critical: it tells Terraform not to start creating compute VMs until the master VM exists and its floating IP is attached. The compute cloud-init script polls for NFS from the master. If the master VM did not exist yet when the compute node started polling, the poll would time out after 5 minutes and the bootstrap would fail.

### 4.6 `templates/master_init.sh.tpl`

This is the cloud-init bootstrap script for the master node. It runs as root during the first boot of the VM. The log of everything it does is written to `/var/log/cluster_init.log`.

**Step 1 — Hostname and `/etc/hosts`**

Sets the system hostname to `master` and writes every cluster node's IP–hostname mapping into `/etc/hosts`. This is necessary because the internal network has no DNS server — all name resolution uses `/etc/hosts`.

```bash
hostnamectl set-hostname master
cat >> /etc/hosts << 'HOSTS'
192.168.100.10  master
192.168.100.11  node01
192.168.100.12  node02
HOSTS
```

**Step 2 — Package installation**

Installs all required software via `apt-get`:
- `nfs-kernel-server`: NFS server daemon
- `slurm-wlm`: the full SLURM suite (includes `slurmctld`, `slurmd`, and CLI tools)
- `munge`: the authentication daemon required by SLURM
- `openmpi-bin` + `libopenmpi-dev`: MPI runtime for parallel programs

**Step 3 — NFS server setup**

Creates and exports the shared filesystem:

```bash
mkdir -p /shared
chmod 1777 /shared
echo "/shared 192.168.0.0/16(rw,sync,no_root_squash,no_subtree_check)" >> /etc/exports
systemctl enable --now nfs-kernel-server
exportfs -ra
```

The export options: `rw` (read-write), `sync` (writes are synchronous — safer), `no_root_squash` (root on a client maps to root on the server, needed to install config files), `no_subtree_check` (avoids issues with files being renamed). The `/16` CIDR is intentionally broad to cover any subnet variation without requiring export updates.

**Step 4 — Munge key generation and distribution**

Munge authenticates SLURM daemon communication. All nodes must share an identical key. The master generates it and places it in `/shared` for compute nodes to retrieve:

```bash
dd if=/dev/urandom bs=1 count=1024 2>/dev/null | base64 -w 0 | head -c 1024 \
    > /etc/munge/munge.key
chmod 0400 /etc/munge/munge.key
chown munge:munge /etc/munge/munge.key
install -m 0400 /etc/munge/munge.key /shared/munge.key
systemctl enable --now munge
```

**Step 5 — `slurm.conf` generation**

The SLURM configuration file is written directly by the template. All node-specific values (IPs, CPU counts, RAM) come from Terraform variables, so the config is always consistent with the actual infrastructure:

```
ClusterName=mhpc
SlurmctldHost=master(192.168.100.10)
AuthType=auth/munge
ReturnToService=2
NodeName=node01 NodeAddr=192.168.100.11 CPUs=2 RealMemory=3500 State=UNKNOWN
NodeName=node02 NodeAddr=192.168.100.12 CPUs=2 RealMemory=3500 State=UNKNOWN
PartitionName=compute Nodes=node[01-02] Default=YES MaxTime=INFINITE State=UP
```

`ReturnToService=2` is especially important in cloud environments: it tells `slurmctld` to automatically return a node to `idle` after it reconnects, even if it was previously marked `down`. Without this, every VM reboot requires manual `scontrol update nodename=... state=resume`.

`slurm.conf` and `cgroup.conf` are then copied to `/shared` so compute nodes can retrieve them.

**Step 6 — Starting SLURM services**

Both `slurmctld` (the controller) and `slurmd` (the worker daemon) are started on the master. This means the master can also accept and run jobs, which is useful for testing.

**Step 7 — Shared user creation**

Creates a script `/shared/setup_user.sh` that adds a `hpcuser` account with a fixed UID (2000), then executes it on the master. Compute nodes will also run this script. The fixed UID matters because NFS maps file ownership by UID number, not by username. If `hpcuser` had UID 2001 on node01 but UID 2003 on node02, file ownership would appear inconsistent across the cluster.

### 4.7 `templates/compute_init.sh.tpl`

This script runs on each compute node. It is simpler than the master script because compute nodes are consumers of cluster services, not providers.

**Steps 1–2** are identical in purpose to the master: set the hostname (`node01`, `node02`, ...) and populate `/etc/hosts` with every node's address.

**Step 3 — Wait for NFS**

The most important part of the compute bootstrap:

```bash
MAX_WAIT=300
WAITED=0
until showmount -e 192.168.100.10 2>/dev/null | grep -q "/shared"; do
    sleep 5
    WAITED=$((WAITED + 5))
done
```

The compute node polls `showmount` every 5 seconds until it sees the master's NFS export. This is necessary because all VMs boot simultaneously and the master needs 2–5 minutes to install packages and start NFS. The timeout is 300 seconds. See section 6 for a full discussion of why this polling exists.

**Steps 4–6** mount `/shared`, then copy `munge.key` and `slurm.conf` from the shared filesystem into their proper local locations. Each file has its own polling loop because the master writes these files after starting NFS, with a short delay.

**Steps 7–8** start `slurmd` and create the `hpcuser` account with the same fixed UID as on the master.

## 5. The Template System Explained

Terraform's `templatefile()` function connects Terraform's knowledge of the infrastructure to the shell scripts that configure the VMs. It works in two stages:

**Stage 1 — at `terraform apply` time**: Terraform reads the `.tpl` file, substitutes all `${variable}` placeholders with their computed values, and evaluates all `%{ for ... }` loops to generate repeated lines. The result is a plain bash script with no template syntax remaining. This rendered script is what OpenStack receives as `user_data`.

**Stage 2 — at VM first boot**: cloud-init reads the `user_data` script and executes it as root.

The template syntax uses two distinct mechanisms:

| Syntax | Meaning |
|---|---|
| `${variable}` | Substituted by Terraform at plan/apply time |
| `$${variable}` | Becomes `${variable}` in the final script (escaped, for bash variable expansion) |
| `%{ for i, ip in list ~}` | Terraform loop — generates repeated output |
| `%{ endfor ~}` | End of loop; `~` strips the trailing newline |

**Example**: for `compute_node_count = 2` and `node_ips = ["192.168.100.11", "192.168.100.12"]`, this template block in `master_init.sh.tpl`:

```
%{ for i, ip in node_ips ~}
NodeName=node0${i + 1} NodeAddr=${ip} CPUs=${compute_cpus} RealMemory=${compute_mem_mb} State=UNKNOWN
%{ endfor ~}
```

is rendered by Terraform into:

```
NodeName=node01 NodeAddr=192.168.100.11 CPUs=2 RealMemory=3500 State=UNKNOWN
NodeName=node02 NodeAddr=192.168.100.12 CPUs=2 RealMemory=3500 State=UNKNOWN
```

This means `slurm.conf` is always correct regardless of the value of `compute_node_count`. Change it to 4, run `terraform apply`, and the configuration adapts automatically without editing any template.

## 6. The Chicken-and-Egg Problem: `munge.key`

SLURM requires that every daemon — `slurmctld` on the master and every `slurmd` on every compute node — authenticates its communication with munge. Munge works by verifying that all communicating processes share the same secret key stored at `/etc/munge/munge.key`. This key must be:

- **randomly generated** at cluster creation time (a static key in version control is a security risk)
- **identical on all nodes** (a different key on any node means that node cannot communicate with SLURM)

In a traditional HPC cluster, a sysadmin generates the key once and distributes it manually or via configuration management. With cloud provisioning, all VMs boot simultaneously and the compute nodes need the key before they can start `slurmd`.

This project solves the problem using the shared NFS filesystem as a staging area:

```
master boots
  → installs packages
  → starts NFS server, exports /shared
  → generates /etc/munge/munge.key (random)
  → copies it to /shared/munge.key
  → starts munge, slurmctld, slurmd

compute01 boots (in parallel with master)
  → installs packages
  → polls showmount until master's NFS appears  ← waits here 2-5 min
  → mounts /shared
  → polls until /shared/munge.key exists        ← waits here ~30 sec
  → copies /shared/munge.key to /etc/munge/munge.key
  → polls until /shared/slurm.conf exists
  → copies /shared/slurm.conf to /etc/slurm/slurm.conf
  → starts munge, slurmd
```

The compute node is patient: it waits up to 5 minutes for each step. This is more than enough time for any OpenStack environment. If the wait exceeds the timeout, the script prints an error and exits with a non-zero status, which cloud-init records in the log.

## 7. Prerequisites

**Software on your local machine**:

- **Terraform** ≥ 1.5 or **OpenTofu** ≥ 1.6 — both work identically with this project
- **OpenStack CLI** — `pip install python-openstackclient` — needed to look up resource names
- **SSH key pair** — you need `~/.ssh/id_rsa` and `~/.ssh/id_rsa.pub`

```bash
# Verify Terraform
terraform version

# Verify OpenStack CLI
openstack --version

# Verify SSH key exists
ls ~/.ssh/id_rsa.pub

# If you need to generate one:
ssh-keygen -t ed25519 -C "your.name@example.com"
# then set public_key_path = "~/.ssh/id_ed25519.pub" in terraform.tfvars

## 8. Configuration: Adapting to Your OpenStack

Every OpenStack deployment has slightly different resource names. Before deploying, identify the correct names in your project.

```

```

# Note its exact Name, VCPUs, and RAM values
```

**Step 5 — Create your configuration file**:
```bash
cp terraform.tfvars.example terraform.tfvars
```

Edit `terraform.tfvars`:

cluster_name           = "mhpc"

## 11. Running Jobs with SLURM

### Interactive execution

```bash
# Run one command on a single node
srun -N 1 hostname

# Run on all compute nodes simultaneously
srun -N 2 hostname

# Open an interactive shell on one compute node
srun -N 1 --pty bash
# You are now inside a shell on node01 (or node02)
# Type 'exit' to return to the master
```

### Batch job submission

Write a job script and submit it with `sbatch`. Job output goes to the file specified by `--output`.

```bash
cat > /shared/my_job.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=hello
#SBATCH --nodes=2
#SBATCH --ntasks=4
#SBATCH --output=/shared/hello_%j.out
#SBATCH --error=/shared/hello_%j.err

echo "Job $SLURM_JOB_ID started at $(date)"
echo "Running on: $SLURM_NODELIST"
srun hostname
EOF

sbatch /shared/my_job.sh

squeue           # watch the queue
cat /shared/hello_*.out   # read results when done
```

### MPI parallel jobs

```bash
# Compile
cat > /shared/hello_mpi.c << 'EOF'
#include <mpi.h>
#include <stdio.h>
int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);
    int rank, size; char host[256];
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);
    gethostname(host, sizeof(host));
    printf("Hello from rank %d/%d on %s\n", rank, size, host);
    MPI_Finalize();
}
EOF
mpicc -o /shared/hello_mpi /shared/hello_mpi.c

# Submit MPI job: 2 nodes, 4 total processes (2 per node)
cat > /shared/mpi_job.sh << 'EOF'
#!/bin/bash
#SBATCH --job-name=mpi_hello
#SBATCH --nodes=2
#SBATCH --ntasks=4
#SBATCH --ntasks-per-node=2
#SBATCH --output=/shared/mpi_%j.out
srun /shared/hello_mpi
EOF

sbatch /shared/mpi_job.sh
cat /shared/mpi_*.out
```

### Essential SLURM commands

| Command | Purpose |
|---|---|
| `sinfo` | Show partition and node state |
| `squeue` | Show the job queue |
| `sbatch job.sh` | Submit a batch job |
| `srun -N 2 cmd` | Run `cmd` interactively on 2 nodes |
| `scancel <jobid>` | Cancel a job |
| `scontrol show job <id>` | Detailed job information |
| `scontrol show node node01` | Detailed node information |
| `sacct` | Job accounting history |

## 12. Accessing Compute Nodes

Compute nodes have no floating IP and are not reachable directly from the internet. Use the master as an SSH jump host:

```bash
# Direct jump host syntax
ssh -J ubuntu@<FLOATING_IP> ubuntu@node01
ssh -J ubuntu@<FLOATING_IP> ubuntu@node02
```

To avoid typing this every time, add an entry to `~/.ssh/config`:

```
Host mhpc-master
    HostName <FLOATING_IP>
    User ubuntu
    IdentityFile ~/.ssh/id_rsa

Host node01 node02
    User ubuntu
    ProxyJump mhpc-master
    IdentityFile ~/.ssh/id_rsa
```

After this, `ssh node01` works from your laptop without specifying the jump host.

From the master itself, you can SSH to compute nodes directly by hostname (because `/etc/hosts` has the mappings):

```bash
# From master:
ssh node01
ssh node02
```

## 13. Troubleshooting

### Nodes appear as `down` or `unknown` in `sinfo`

This usually means `slurmd` on a compute node is not connected to `slurmctld`. Work through these steps in order:

```bash
# 1. Check cloud-init completed on the compute node
ssh node01 "sudo tail -30 /var/log/cluster_init.log"
# Look for "Bootstrap node01 completato" at the end

# 2. Check slurmd is running
ssh node01 "systemctl status slurmd"

# 3. Check munge authentication works
ssh node01 "munge -n | unmunge"
# Should say: STATUS: Success (0)

# 4. Manually bring nodes back to idle
sudo scontrol update nodename=node01 state=resume
sudo scontrol update nodename=node02 state=resume

# 5. Check slurmctld logs for errors
sudo tail -50 /var/log/slurm/slurmctld.log
sudo journalctl -u slurmctld -n 100 --no-pager
```

### Cloud-init is still running

`terraform apply` finishes before cloud-init completes. If you SSH in too soon, the cluster is not ready yet. Follow the progress:

```bash
# Master
ssh ubuntu@<FLOATING_IP> "sudo tail -f /var/log/cluster_init.log"

# Compute nodes (via jump host)
ssh -J ubuntu@<FLOATING_IP> ubuntu@node01 "sudo tail -f /var/log/cluster_init.log"
```

Wait for the "Bootstrap completato" message on all nodes.

### NFS not mounted on compute nodes

```bash
# Check from the compute node
ssh node01 "showmount -e 192.168.100.10"
# Should show: /shared  192.168.0.0/16

ssh node01 "df -h /shared"
# Should show the NFS filesystem mounted

# If not mounted, retry:
ssh node01 "sudo mount -a"
```

If `showmount` returns nothing, the NFS server may not be running on the master:
```bash
sudo systemctl status nfs-kernel-server
sudo exportfs -v
```

### Munge key mismatch

```bash
# The checksums must be identical on all nodes
md5sum /etc/munge/munge.key
ssh node01 "sudo md5sum /etc/munge/munge.key"
ssh node02 "sudo md5sum /etc/munge/munge.key"

# If they differ, copy the correct key from /shared:
ssh node01 "
  sudo install -m 0400 -o munge -g munge /shared/munge.key /etc/munge/munge.key
  sudo systemctl restart munge slurmd
"
sudo systemctl restart slurmctld
```

### `terraform apply` fails with authentication errors

```bash
# Verify credentials are loaded
echo $OS_AUTH_URL
echo $OS_USERNAME

# Reload if empty
source ~/openrc.sh
```

### SSH: "Permission denied (publickey)"

Make sure you are using the correct private key and username:

```bash
ssh -i ~/.ssh/id_rsa ubuntu@<FLOATING_IP>
```

Ubuntu 22.04 cloud images use `ubuntu` as the default username, not `root` or your OpenStack account name.

## 14. Destroying the Cluster

When you are done, destroy all resources to free your cloud quota:

```bash
source ~/openrc.sh
terraform destroy
```

Terraform lists everything it will delete and asks for confirmation. Type `yes`. All VMs, ports, floating IPs, security groups, subnet, router, and the virtual network are removed. Nothing persists — the cluster is fully ephemeral.

To redeploy at any time, simply run `terraform apply` again. A fresh cluster is created from scratch in minutes.

## 15. Key Design Decisions

Understanding why the code is written the way it is helps you adapt it and diagnose problems.

**Security groups on ports, not instances.** The OpenStack Nova API has a known bug: security groups attached directly to `openstack_compute_instance_v2` are not applied at creation time, only after a state refresh. Attaching them to `openstack_networking_port_v2` resources works correctly and immediately. This is the recommended approach for all OpenStack releases after Kilo (2015).

**Fixed IPs via Neutron ports.** `slurm.conf` contains literal IP addresses for each compute node (`NodeAddr=192.168.100.11`). Dynamic DHCP addressing would break SLURM every time a node rebooted and happened to get a different IP. Neutron ports with `fixed_ip` reserve the address permanently in the subnet, ensuring stability across reboots.

**No floating IPs on compute nodes.** Only the master is exposed to the internet. Compute nodes are reachable only through the master. This reduces the attack surface and mirrors the standard architecture of production HPC clusters, where users log in to a designated login node and never SSH directly to compute nodes.

**Credentials via environment variables.** The `provider "openstack" {}` block is empty. Terraform reads `OS_AUTH_URL`, `OS_USERNAME`, and other credentials from the shell environment populated by `source openrc.sh`. This is the standard and safe authentication pattern — credentials never appear in `.tf` files, which could be committed to version control.

**`munge.key` distributed via NFS.** The shared filesystem (`/shared`) exists and is mounted before `slurmd` starts on compute nodes. Using it as a staging area for the key is simpler than alternatives (pre-built image, object storage, secrets manager) and keeps the architecture entirely self-contained.

**`ReturnToService=2` in `slurm.conf`.** Cloud VMs can be rebooted by the hypervisor for maintenance or by the OS for updates. Without this setting, every reboot puts the node in permanent `down` state requiring manual `scontrol update nodename=... state=resume`. With `ReturnToService=2`, `slurmctld` automatically returns a reconnecting node to `idle`. This is essential for any cluster you do not monitor 24/7.

**`depends_on` between compute and master VMs.** Terraform's dependency resolution normally relies on resource attribute references. Compute VMs reference only the network port, not any attribute of the master VM. Without an explicit `depends_on`, Terraform would create all VMs simultaneously, and compute nodes would start polling for NFS before the master had even booted.

## 16. Exercises and Extensions

Once the basic cluster is running, these are good directions to explore.

**Exercise 1: Scale the cluster.** Change `compute_node_count = 4` in `terraform.tfvars` and run `terraform apply`. Observe that Terraform adds only the two new compute nodes and ports, without touching the master or existing nodes. Verify `sinfo` shows four compute nodes.

**Exercise 2: Inspect `slurm.conf`.** Read `/etc/slurm/slurm.conf` on the master. Trace each directive back to the Terraform variable or template expression that generated it. Compare with `/shared/slurm.conf` — they should be identical. Why does the compute node read from `/shared` rather than waiting for the file to be placed directly at `/etc/slurm/slurm.conf`?

**Exercise 3: Study the cloud-init log.** Read `/var/log/cluster_init.log` on both master and node01. Identify timestamps for each phase: package installation, NFS start, munge start, SLURM start. How long did node01 wait for NFS? What would happen if `MAX_WAIT` were set to 30 seconds instead of 300?

**Exercise 4: Intentionally break and fix munge.** On node01, overwrite the munge key with random bytes:
```bash
ssh node01 "sudo dd if=/dev/urandom bs=1 count=1024 > /etc/munge/munge.key && sudo systemctl restart munge slurmd"
```
What does `sinfo` show? What does the slurmctld log report? Restore the correct key and bring the cluster back to a healthy state without rerunning `terraform apply`.

**Exercise 5: Add a shared home directory.** Modify `master_init.sh.tpl` to also export `/home/hpcuser` via NFS, and modify `compute_init.sh.tpl` to mount it at `/home/hpcuser`. After redeploying, files created by `hpcuser` in their home directory on any node will be visible on all others — the standard HPC pattern for user data.

**Exercise 6: Change the compute flavor.** Update `compute_flavor`, `compute_cpus`, and `compute_mem_mb` in `terraform.tfvars`. Run `terraform plan` and observe that Terraform plans to destroy and recreate the compute VMs (because flavor is immutable on a running instance). Run `terraform apply`. After the compute nodes reboot and reconnect, verify `scontrol show node node01` reports the new resource counts.
